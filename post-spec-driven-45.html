45% menos latencia: un caso de Spec-Driven Development

Cómo escribir la especificación antes que el código me ayudó a diagnosticar mejor un backend PHP lento — y cambió la manera en que ataco cualquier optimización.

12 · JUL · 2026 · 6 min de lectura · Publicado

Hace unos meses heredé un backend en PHP que estaba tardando alrededor de 800 ms en responder los endpoints de mayor tráfico. El equipo lo sentía "lento", los usuarios abandonaban antes de completar acciones y cada semana aparecía una queja nueva.

La reacción instintiva es abrir el profiler y empezar a tocar. Pero antes de escribir una línea, escribí una especificación. Este es el proceso —Spec-Driven Development— y por qué me sirvió para bajar los tiempos de carga un 45%.

SDD en una frase: escribir la especificación funcional como artefacto principal —alcance, contratos de datos, casos borde y criterios de aceptación verificables— antes de la implementación.
El flujo: Especificar → Planificar → Tareas → Implementar.

1. Especificar

Antes de mirar el código heredado, escribí lo que un endpoint "rápido" tenía que cumplir. Cuatro líneas, pero criterios verificables:

GET /api/orders?status=pending
  - Responde en < 200 ms para hasta 50.000 registros
  - Devuelve cliente e items en una sola respuesta
  - Máximo 5 queries por request
  - Paginado (50 por página)

Cuando refactorice, no voy a sentir si mejoró — lo voy a medir contra esos números. Ese es el punto.

Si no puedo escribir el criterio de aceptación, no entiendo el problema. Y si no entiendo el problema, cualquier cosa que codee va a ser adivinar.

2. Planificar

Con la spec en mano medí el estado actual con DB::listen() y el profiler. Salieron tres candidatos claros:

  • N+1 clásico. Por cada orden, una consulta extra a customers y otra a order_items. Más de 200 queries por request.
  • Índice ausente. La tabla orders no tenía índice compuesto sobre (status, created_at), que era justamente el filtro y el orden principal.
  • Respuesta sobrecargada. Se serializaba la orden completa cuando el frontend solo pintaba cuatro campos.

3. Tareas

De esos candidatos salieron tres tareas concretas, priorizadas por impacto / esfuerzo:

  1. Eager loading para eliminar el N+1.
  2. Índice compuesto en (status, created_at).
  3. DTO ligero para la respuesta.

4. Implementar

Solo aquí toqué código. El "antes" era casi una postal del anti-patrón:

// Antes
$orders = Order::where('status', 'pending')->get();
foreach ($orders as $order) {
    $data[] = [
        'id'          => $order->id,
        'customer'    => $order->customer->name,    // 1 query extra
        'items_count' => $order->items->count(),   // 1 query extra
    ];
}

Y el "después":

// Después
$orders = Order::with(['customer:id,name'])
    ->withCount('items')
    ->where('status', 'pending')
    ->orderBy('created_at', 'desc')
    ->paginate(50);

Más el índice, que fue lo que más movió la aguja:

CREATE INDEX idx_orders_status_created
    ON orders (status, created_at DESC);

Resultados

MétricaAntesDespués
Tiempo total820 ms450 ms
Queries por request2033
Uso de memoria48 MB14 MB

El 45% de mejora vino de dos decisiones simples: eager loading y un índice bien pensado. Nada exótico.

Qué me llevo

  • Diagnosticar antes de refactorizar. La spec me obliga a describir el objetivo antes de tocar código, y eso descarta rutas equivocadas.
  • Los tres candidatos no fueron adivinanzas. Salieron de comparar el comportamiento actual contra los criterios escritos.
  • El código que no escribes importa tanto como el que sí. La spec me evitó cachear "por si acaso" un endpoint que en realidad solo necesitaba un índice.

Trabajo así todos los días — especificar, planificar, tareas, implementar— sea código a mano o asistido por IA. Cuando el punto de partida está bien escrito, el resto se parece bastante a completar un formulario.
